Project Review · PI Cockpit

SDK Integration Plan Survives Peer Review — Three Corrections Needed Before Code

The plan to wire PI's TypeScript SDK directly into the Cockpit hub is architecturally sound, but string-based model IDs, session switching, and typedef verification need attention.

By {{AGENT_NAME}} Research · Session: Claude Opus 4.7 · 1M ctx

ARLINGTON, VA —The PI Cockpit project — which embeds a full AI control surface into Obsidian via the Custom Frames plugin — crossed a key checkpoint this week. A plan to replace the current pi-acp bridge with a direct SDK integration survived a thorough review, but three concrete corrections must land before any agent writes hub/pi-bridge.js.

The review verified four "check before coding" items against the installed @mariozechner/pi-coding-agent@0.73.1 package. The verdict: the package name is correct, dependencies are populated in the worktree, and core APIs — createAgentSession, AgentSession, ModelRegistry — exist as documented. The worktree at ~/.worktrees/{{AGENT_NAME}}-pi-cockpit/ is ready for development.

Correction 1: Models aren't strings

The original plan assumed session.setModel("deepseek-v4-pro") would work. It won't. The SDK expects Model<any> objects, not string IDs. The bridge needs to import getModel from @mariozechner/pi-ai and map widget model IDs to (provider, modelId) pairs before passing to the SDK.

Correction 2: Session switching is multi-step

The plan described a single piBridge.switchSession(path) call. In reality, switching requires tearing down the current AgentSession, building a new SessionManager against the target path, calling createAgentSession again, and re-subscribing to events. The AgentSessionRuntime class may simplify this — the implementing agent should read its typedef before deciding.

Correction 3: Read typedefs, not blog posts

The plan cited a Nader Dabit article for API shapes. The review flagged this as risky — typedefs are the canonical source. Three files must be read before coding: agent-session.d.ts (event shapes), agent-session-runtime.d.ts (lifecycle), and model-registry.d.ts (model API).

What the plan got right

The decision to use direct SDK over pi-acp was correct — the MCP and skills passthrough issues on the acp path would have been problematic. Skills and MCP display widgets don't need changes (they read from the filesystem). The WebSocket protocol stays intact — only hub-side handlers change. The plugin ItemView structure remains.

The review also surfaced an operational detail: the hub runs from the worktree, not the vault. During active development, the launchd-managed hub should be killed and the dev loop run manually from ~/.worktrees/{{AGENT_NAME}}-pi-cockpit/pi-cockpit/hub/ to avoid the styles.css revert issue that bit the team earlier.

Reference · Agent Architecture

Nader Dabit Maps the PI Agent Stack: Four Layers From Provider to Terminal

A comprehensive guide by developer-educator Nader Dabit walks through PI's composable TypeScript architecture, demonstrating how four independent layers compose into a production-grade coding agent without vendor lock-in.

The stack begins with pi-ai, which normalizes LLM calls across providers — the foundational layer that makes provider switching possible without code changes. Above it sits pi-agent-core, handling the agent loop, tool execution, and event streaming. This is the heart of the agent — the stateful engine that manages turns, messages, and tool calls.

The pi-coding-agent layer adds file-system tools — read, write, edit, bash — and session persistence, turning the core agent into a full coding assistant. Finally, pi-tui provides the terminal UI with markdown rendering, code blocks, and an inline editor. The article demonstrates each layer in isolation, showing teams how to swap or replace components without breaking the whole system.

Analysis · The Filesystem Pattern

Why the Vault Architecture Works: Zero-Build Artifact Delivery

The {{AGENT_NAME}} system — and by extension the PI Cockpit — rests on an architectural insight that's easy to miss: agents write files, watchers detect changes, surfaces re-render. There is no database, no API layer, no deployment step.

This pattern was first described in the Agent Surface issue of this publication, but it's worth restating in light of the Cockpit review. The hub's skill scanner uses chokidar to watch the filesystem for skill changes. The artifact dashboard watches 6-Agent/tma/ for new HTML files. The session monitor reads PI session files as they're written. Everything is a file.

This is not a compromise — it's a deliberate choice. The filesystem is faster than any REST call, requires no authentication, and works with every tool in the ecosystem. When the PI SDK bridge lands, it will join this pattern: the hub writes session state, the SDK consumes it, the widgets react. No database required.
