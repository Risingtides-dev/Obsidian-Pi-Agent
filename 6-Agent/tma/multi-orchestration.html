Systems architecture

Linear agent workflows in a multi-orchestration pipeline

How to chain Slack, Linear, GitHub, and local agent runtimes into a continuous loop — orchestrator builds PRDs, workers execute tasks, reviewer agents gate merges, and status flows back to the team.

45 min read · may 12 2026

01 — Architecture

Four platforms, four agent types, one event loop

The multi-orchestration pattern connects four platforms through a shared event loop. Each platform runs its own agent. Events propagate across them via webhooks and API calls, creating a continuous workflow where human review is interleaved with agent execution at every stage.

The orchestrator lives where the conversation happens (Slack). It translates human intent into structured tasks (Linear), which worker agents pick up and execute locally. Output flows back through pull requests (GitHub), where reviewer agents provide automated feedback. The orchestrator closes the loop by reporting results to the channel.

Layer 1

Slack

Orchestrator agent reads channel history, builds PRDs, decomposes to Linear issues

Layer 2

Linear

Task backbone — issues assigned to workers, webhooks trigger execution, GitHub integration auto-links PRs

Layer 3

GitHub

Webhooks fire on PR events, reviewer agent analyzes diffs, check runs gate merges

Layer 4

Local runtimes

Worker agents poll Linear, execute in worktrees, push branches — filesystem is the protocol

Core insight

Each platform already has agent primitives. The innovation is chaining them through webhooks rather than building a centralized orchestrator. The events are the architecture.

Event flow

Slack msg → Orchestrator parses intent → Creates Linear issue
Linear webhook → Worker picks up task → Writes code locally
Worker pushes branch → Opens PR → GitHub webhook fires
Reviewer agent analyzes diff → Posts review → Check run created
Human merges → Linear auto-closes → Orchestrator posts to Slack

Linear and github integration (built-in)

Linear natively integrates with GitHub. When a branch or PR references an issue ID, they link automatically. For agent workflows, this means a worker creates a branch named ENG-123-add-oauth-flow, pushes it, and Linear links it to the issue without any API call.

→ Auto-linking — include ENG-123 in branch name or pr description

→ Status sync — pr merge or close updates linear issue state

→ Branch creation — linear can create branches via github api

Oauth apps versus github apps

Two authentication models exist. For agent workflows, use GitHub Apps — granular permissions, per-installation tokens, built-in webhooks, higher rate limits. OAuth apps act as the authenticated user and inherit their permissions, which is simpler but less appropriate for automated agents.

Reviewer agent pattern

The reviewer agent listens for pull_request.opened webhooks and executes a structured pipeline.

Reviewer agent — on pull_request.opened

1. receive webhook → extract pr number, head sha, repo
2. fetch pr diff → GET /repos/:owner/:repo/pulls/:number/files
3. fetch pr description + linked linear issue (regex branch name)
4. run analysis → diff → agent prompt → structured review
5. post review → POST /repos/:owner/:repo/pulls/:number/reviews
     { "body": "...", "event": "COMMENT" | "REQUEST_CHANGES" | "APPROVE",
       "comments": [{"path": "src/auth.ts", "line": 42, "body": "..."}] }
6. update linear → post comment with review summary + pr link
7. signal orchestrator → post review status to slack thread

Check runs as quality gates

GitHub apps can create check runs that appear in the pr ci panel. A reviewer agent creates a check run named agent-review that passes or fails based on analysis, with line-level annotations.

POST /repos/:owner/:repo/check-runs
{
  "name": "Agent Review",
  "head_sha": "abc123...",
  "status": "completed",
  "conclusion": "success" | "failure" | "neutral",
  "output": {
    "title": "3 suggestions, 0 blockers",
    "summary": "...",
    "annotations": [
      { "path": "src/auth.ts", "start_line": 42,
        "annotation_level": "warning", "message": "..." }
    ]
  }
}

Combined with branch protection rules (require agent-review check before merge), this creates a hard gate — no code passes without agent review.

Step 3 — webhook receiver (local)

The webhook receiver runs locally on the mac mini and is exposed via cloudflare tunnel.

// webhook-receiver.ts — runs as launchd daemon
import { Webhooks } from "@octokit/webhooks";
import { createNodeMiddleware } from "@octokit/webhooks";
import { createServer } from "node:http";

const webhooks = new Webhooks({ secret: process.env.WEBHOOK_SECRET });

webhooks.on("pull_request.opened", async ({ payload }) => {
  const { number, head } = payload.pull_request;
  const issueId = extractLinearIssue(head.ref); // regex ENG-XXX

  // trigger reviewer agent
  await triggerReviewerAgent({ prNumber: number, issueId });
});

webhooks.on("pull_request.closed", async ({ payload }) => {
  if (payload.pull_request.merged) {
    await updateLinearIssue(
      extractLinearIssue(payload.pull_request.head.ref),
      { stateId: "done-state-id" }
    );
  }
});

createServer(createNodeMiddleware(webhooks)).listen(3000);

Step 4 — worker agent loop

Worker agents run as pi sessions in worktrees, polling linear for assigned tasks.

// worker-loop.ts — runs every 60s via cron or while+sleep
async function workerLoop(workerId: string) {
  const issues = await linearGraphQL(`
    query {
      issues(filter: {
        assignee: { name: { eq: "${workerId}" } },
        state: { name: { eq: "Todo" } }
      }) { nodes { id title description branchName } }
    }
  `);

  for (const issue of issues) {
    // 1. mark in progress
    await markInProgress(issue.id);

    // 2. create worktree + branch
    exec(`cd ~/dev/{{AGENT_NAME}} && wt ${issue.branchName}`);

    // 3. spawn pi coding agent with issue context
    const session = createAgentSession({
      systemPrompt: buildPrompt(issue.description),
      cwd: `~/.worktrees/{{AGENT_NAME}}-${issue.branchName}`
    });
    await session.prompt(`Implement: ${issue.title}`);

    // 4. commit, push, open pr
    exec(`git add . && git commit -m "${issue.title}"`);
    exec(`git push origin ${issue.branchName}`);
    exec(`gh pr create --title "${issue.title}" --body "Closes ${issue.branchName}"`);

    // 5. mark ready for review
    await setInReview(issue.id);
  }
}

Step 5 — orchestrator (slack bot)

The orchestrator runs as a slack bot using @slack/bolt.

// orchestrator.ts
import { App } from "@slack/bolt";

const app = new App({
  token: process.env.SLACK_BOT_TOKEN,
  signingSecret: process.env.SLACK_SIGNING_SECRET
});

app.event("app_mention", async ({ event, say, client }) => {
  const thread = await client.conversations.replies({
    channel: event.channel,
    ts: event.thread_ts || event.ts
  });

  // build prd from conversation context
  const prd = await buildPRD(thread.messages, event.text);
  await say({ text: prd, thread_ts: event.ts });
});

app.command("/decompose", async ({ command, ack, say }) => {
  await ack();
  const issues = await decomposeToLinear(command.text);
  await say(`Created ${issues.length} issues in linear`);
});
