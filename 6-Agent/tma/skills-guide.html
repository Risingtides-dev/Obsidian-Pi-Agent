Building reusable skills for claude code

How to create custom slash commands that give agents environment variables, database access, and domain context — so they execute repetitive tasks without being told the setup every time.

25 min read · grounded in risingtides campaign hub

01 — Concept

What claude code skills actually are

A claude code skill is a markdown file in .claude/commands/ that gets loaded as a slash command. When you type /my-skill, claude reads the file, inserts its contents into the system prompt, and executes with whatever context, tools, and instructions you've defined.

Skills solve the context problem. Instead of pasting database schemas and environment setup at the start of every session, you encode it once in a file. The agent arrives already knowing the tables, the variables, the business rules, and the output format you expect.

The same pattern works in pi via .pi/skills/ — the html-artifacts skill you use is built this way. Both systems treat the markdown file as the skill definition. The difference is in how environment variables are injected.

Claude code skills

Custom slash commands stored as .claude/commands/skill-name.md. Loaded at session start or invoked with /skill-name. Environment variables come from .claude/settings.json.

Pi skills

Markdown files with yaml frontmatter in .pi/skills/skill-name/SKILL.md. Can include style guides, reference files, and tool definitions. Environment comes from pi extensions.

02 — Anatomy

The three layers every skill needs

A well-built skill has three distinct sections. Each answers a different question the agent will ask itself when it starts the task.

Layer 1

Identity and purpose — what am i doing and why

The first section tells the agent its role and what output is expected. Be specific about the domain, the audience, and the format. This is where you kill ambiguity.

# Campaign Intelligence Agent

You are a data analyst agent for Rising Tides, a social media
marketing agency running TikTok and Instagram UGC campaigns for
record labels. Your job is to answer questions about campaigns,
creators, sound IDs, and performance data.

## Output rules
- Always return data in a markdown table
- Include campaign slug in every response for traceability
- Round dollar amounts to the nearest dollar
- Flag creators with posts_owed > posts_done as behind schedule
- Never expose database credentials or connection strings

Layer 2

Domain context — what does the agent need to know

This is the schema, the business rules, the constraints. Every table, every relationship, every convention. The agent should not have to discover the database structure — you tell it upfront.

## Database Schema (PostgreSQL on Railway)

### campaigns — one row per client campaign
| Column | Type | Description |
|--------|------|-------------|
| id | integer | Primary key |
| slug | varchar | URL-safe identifier (e.g. alameda_broken_record_promo) |
| title | varchar | Human-readable name |
| artist | varchar | Recording artist name |
| song | varchar | Track title |
| sound_id | varchar | TikTok sound ID (the audio the creator uses) |
| additional_sounds | jsonb | Array of extra sound IDs for multi-sound campaigns |
| status | varchar | "active" or "finished" |
| budget | float | Total campaign budget in dollars |
| platform | varchar | "tiktok" or "instagram" |
| cobrand_share_url | text | Cobrand tracking dashboard URL |

### creators — one row per booked creator per campaign
| Column | Type | Description |
|--------|------|-------------|
| id | integer | Primary key |
| campaign_id | integer | FK to campaigns.id |
| username | varchar | Creator's social media handle |
| posts_owed | integer | Total posts the creator committed to |
| posts_done | integer | Posts confirmed completed |
| total_rate | float | Total payment for all posts |
| per_post_rate | float | Payment per individual post |
| paypal_email | varchar | Creator's PayPal for payments |
| paid | varchar | "yes" or "no" |
| platform | varchar | "tiktok" or "instagram" |
| niches | jsonb | Array of content categories |

### matched_videos — one row per tracked video post
7640 total rows as of May 2026.

### Key relationships
- creators.campaign_id → campaigns.id
- matched_videos.creator_id → creators.id
- campaigns with completion_status = "completed" are finished
- sound_id values are TikTok audio identifiers (16–19 digit numeric strings)

Layer 3

Tools and environment — how does the agent access the world

This is the connection layer. You tell the agent which tools to use and where the secrets live. For claude code, environment variables are set in .claude/settings.json under env. The agent sees them as regular env vars.

## Environment variables

All set in .claude/settings.json under the "env" key:

DATABASE_URL — PostgreSQL connection string
  Format: postgresql://user:pass@host:5432/db
  For Railway: use railway run or inject via railway variables
  The .internal hostname only works inside Railway's network
  Use railway connect postgres for local querying

ANTHROPIC_API_KEY — Claude API key (for sub-agent calls)

## Tools available

- bash — run psql queries via railway connect or direct connection
- read — read project files (models.py, config.py for schema reference)
- write — never write to the database directly (read-only analysis)

## Query approach

Use railway connect postgres for local development:
  echo "SELECT ..." | railway connect postgres

Or use psql directly if DATABASE_URL has a public host:
  psql "$DATABASE_URL" -c "SELECT ..."

## Current database stats (May 2026)

The live database on happy-wholesome contains:
- 143 campaigns (primarily active, many completed)
- 1,593 creators across all campaigns
- 7,640 matched video posts
- Sound IDs are TikTok audio identifiers (16–19 digit numbers)

03 — Environment

Injecting environment variables into claude code skills

Claude code reads environment variables from .claude/settings.json at the project root. This is where database urls, api keys, and service tokens live. When the agent runs, these are available as regular environment variables.

.claude/settings.json

{
  "env": {
    "DATABASE_URL": "postgresql://postgres:...@postgres.railway.internal:5432/railway",
    "RAILWAY_PROJECT_ID": "1cc43c48-8891-4784-9f72-23174abdd21e",
    "RAILWAY_ENVIRONMENT": "production",
    "ANTHROPIC_API_KEY": "sk-ant-api03-..."
  },
  "enabledPlugins": {
    "claude-code-plugins@claude-code-plugins": true
  }
}

The railway internal hostname problem

Railway sets DATABASE_URL with a .internal hostname that only resolves inside Railway's network. For local development, you have two options.

→ Option a — railway connect (recommended). Use echo "SELECT ..." | railway connect postgres. This tunnels through the railway cli and handles authentication automatically.

→ Option b — public hostname. Replace .internal with the public proxy. Railway exposes a tcp proxy for postgres at a public endpoint. Check your database service settings for the public connection string.

→ Option c — shell wrapper. Write a db-query shell script that wraps railway connect postgres and pass it as an available tool in the skill.

Variable scoping — what the agent can see

Anything in settings.json under env is available to the agent. The agent can also access any variable set in the shell that launched claude code. Be intentional about what you expose.

Variable exposure checklist

→ Database url — needed for queries, never exposed in output

→ Api keys — needed for sub-agent calls, never logged

→ Slack tokens — needed for the orchestrator skill

→ Notion keys — needed for crm sync skill

→ Tiktok cookies — needed for scraping skill, high sensitivity

04 — Best practices

Eight rules for skills that last

1

Skills are read-only instruction files

Don't put executable code in the skill itself. The skill defines what the agent should do and how. The agent writes the code to do it. Separation of specification from execution prevents the skill from becoming a maintenance burden when apis change.

2

Include schema, not just names

Don't just list table names. Include column types, relationships, and real example values. Agents reason better with concrete data shapes than abstract descriptions. Include 1 row of realistic sample data for each table.

3

Define output format explicitly

The difference between a good skill and a bad one is whether you have to reformat the answer. Specify: markdown table, csv, json, or prose. Include an example of the exact output shape you want.

4

Encode constraints, not just capabilities

Tell the agent what not to do. Never modify the database. Never expose credentials. Never query tables you haven't described. The guardrails are as important as the tools.

5

Version the skill alongside the codebase

Skills in .claude/commands/ are part of the repository. They evolve with the schema. When a column is added to the database, update the skill. Git history tells you what the agent knew when.

6

Test with real queries before shipping

Run every example query in the skill against the actual database. A broken example erodes trust faster than no example. Keep a test-queries.sql file that validates the skill's assumptions.

7

One skill, one domain

A skill that can query campaigns, manage creators, sync notion, and post to slack is four skills. Split them. Each skill gets its own markdown file, its own environment variables, and its own clear purpose. Composition happens at the session level when you invoke multiple skills.

8

Include live stats that validate the skill works

Add a section at the bottom with current counts. The agent can quickly verify the database is accessible and the schema matches. If the agent queries SELECT count(*) FROM campaigns and gets 0 instead of 143, it knows something is wrong before wasting a session.

05 — Example

The campaign intelligence skill — built from a real database

Here is a complete, production-ready skill file grounded in the Rising Tides campaign hub database on Railway's happy-wholesome project. The database has 143 campaigns, 1,593 creators, and 7,640 matched videos as of May 2026.

.claude/commands/campaign-intel.md

# Campaign Intelligence Agent

You are a data analyst for Rising Tides, a social media
marketing agency. Your job is to answer questions about
TikTok and Instagram UGC influencer campaigns, creators,
sound IDs, and performance data from the campaign hub
PostgreSQL database on Railway (happy-wholesome).

## Output rules
- Always return data in a markdown table
- Include campaign slug in every response
- Round dollar amounts to nearest dollar
- Flag creators with posts_owed > posts_done as ⚠ behind
- Never expose DATABASE_URL, API keys, or credentials
- Use sentence case for all headings

## Database connection

The database is PostgreSQL on Railway. Connect via:
  echo "SELECT ..." | railway connect postgres

Or use psql:
  psql "$DATABASE_URL" -c "SELECT ..."

DATABASE_URL is set in .claude/settings.json env.

## Schema

### campaigns (143 rows)
| Column | Type | Notes |
|--------|------|-------|
| id | integer | PK |
| slug | varchar | URL-safe (e.g. alameda_broken_record_promo) |
| title | varchar | Human name |
| artist | varchar | Recording artist |
| song | varchar | Track title |
| sound_id | varchar | TikTok audio ID (16-19 digit string) |
| additional_sounds | jsonb | Extra sound IDs |
| status | varchar | "active" or "finished" |
| budget | float | Total budget in USD |
| platform | varchar | "tiktok" or "instagram" |
| cobrand_share_url | text | Cobrand tracking link |
| completion_status | varchar | "none", "booked", or "completed" |

### creators (1,593 rows)
| Column | Type | Notes |
|--------|------|-------|
| id | integer | PK |
| campaign_id | integer | FK to campaigns |
| username | varchar | Social handle |
| posts_owed | integer | Committed posts |
| posts_done | integer | Completed posts |
| total_rate | float | Total payment |
| per_post_rate | float | Per-post payment |
| paypal_email | varchar | Payment destination |
| paid | varchar | "yes" or "no" |
| platform | varchar | "tiktok" or "instagram" |
| niches | jsonb | Content categories |

### matched_videos (7,640 rows)
Link between creators and tracked TikTok/IG videos.

## Key relationships
- creators.campaign_id → campaigns.id
- matched_videos.creator_id → creators.id
- sound_id uniquely identifies TikTok audio for scraping
- campaigns with completion_status = "completed" are finished

## Common queries

Active campaigns with sound IDs:
  SELECT slug, title, sound_id FROM campaigns
  WHERE status = 'active' AND sound_id != '';

Top creators by total payment:
  SELECT c.username, c.total_rate, c.posts_done,
         c.posts_owed, cmp.title
  FROM creators c
  JOIN campaigns cmp ON c.campaign_id = cmp.id
  WHERE c.total_rate > 0
  ORDER BY c.total_rate DESC LIMIT 10;

Creators behind schedule:
  SELECT c.username, c.posts_owed, c.posts_done,
         (c.posts_owed - c.posts_done) as behind,
         cmp.slug
  FROM creators c
  JOIN campaigns cmp ON c.campaign_id = cmp.id
  WHERE c.posts_owed > c.posts_done
  ORDER BY behind DESC;

Campaigns by completion status:
  SELECT completion_status, count(*)
  FROM campaigns GROUP BY completion_status;

## Constraints
- Read-only. Never run INSERT, UPDATE, DELETE, or DROP.
- Never query financial data (paypal_email) unless asked.
- If a query returns 0 results, verify the table name and
  column names against the schema above before trying again.
- The database is production. Timeout queries at 30 seconds.

## Validation (run on first use)
SELECT count(*) FROM campaigns;  -- expect 143
SELECT count(*) FROM creators;   -- expect 1593
SELECT count(*) FROM matched_videos; -- expect 7640

The railway variable bridge

For production use, railway variables are pulled at deploy time. For local development, the railway connect command tunnels through the cli. The pattern for skills is consistent regardless of environment.

# Development (local)
echo "SELECT count(*) FROM campaigns;" | railway connect postgres

# Production (inside railway)
psql "$DATABASE_URL" -c "SELECT count(*) FROM campaigns;"

# The skill should document both approaches so the
# agent chooses the right one based on environment.

07 — Quick start

Create your first skill in five minutes

1

Create the commands directory

mkdir -p .claude/commands
2

Add environment variables to settings

// .claude/settings.json
{
  "env": {
    "DATABASE_URL": "postgresql://..."
  }
}
3

Write the skill file with three layers

// .claude/commands/campaign-intel.md
// Layer 1: identity and output rules
// Layer 2: schema and domain context
// Layer 3: tools and environment access
4

Invoke the skill in claude code

/campaign-intel
Which creators are behind schedule on active campaigns?
Show me their username, posts owed, posts done, and campaign.

A skill is leverage. You spend twenty minutes writing it once, and every future session where you invoke it saves you five minutes of context-setting. After four uses, it pays for itself. After forty, it's changed how you work.
