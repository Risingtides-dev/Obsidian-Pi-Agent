No. 014 / Voice capture system

Voice note
to vault.
No friction.

A Telegram bot accepts native voice notes or uploaded audio. A worker transcribes the file with Whisper, sends the transcript to an LLM, and writes a structured Markdown note into Obsidian.

Target: 6-Agent/tma-mini

01 — Intent

Turn dictated thoughts into durable notes without opening Obsidian.

The human sends a voice memo from Telegram. The system captures source metadata, keeps the raw audio, generates a transcript, summarizes the content, and files the result as Markdown.

The output should be useful later: summary, key takeaways, action items, decisions, open questions, cleaned notes, and raw transcript.

02 — Intake

Path A

Native Telegram voice recorder

Telegram sends a voice object, commonly OGG/Opus. The bot downloads it through getFile, assigns a stable file name, and stores metadata for replay.

Path B

Dropped audio file

Telegram sends an audio or document object. The bot validates MIME type and size, stores the original, and optionally normalizes through ffmpeg.

03 — Output

The Markdown note is the product.

  • Frontmatter. Source, sender, message ID, audio path, transcript path, models, tags.
  • Summary. Three to six bullets for fast recall.
  • Extraction. Tasks, decisions, open questions, people, projects, tools.
  • Cleaned notes. A readable narrative version of the memo.
  • Raw transcript. Preserved for search and audit.

- Do not invent facts, dates, names, tasks, or decisions.

05 — Decisions

Start here

DM-to-bot before channel ingestion.

Direct messages are easier to authorize, test, and debug. A channel can be added later when the capture pipeline is stable.

Preserve source

Keep audio and transcript.

The summary is useful, but the transcript is evidence. Store both so the system can re-summarize later with better prompts.

Reduce hallucination

Uncertainty is a field, not a failure.

The LLM should mark ambiguous items as open questions instead of inventing missing details.
