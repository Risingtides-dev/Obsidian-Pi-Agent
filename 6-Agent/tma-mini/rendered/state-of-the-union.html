Summary

Thoth is an Obsidian vault that doubles as an agent platform. The vault is the filesystem, the notes are the database, and daemons plus PI Cockpit bridge it to phones, browsers, and coding sessions. The architecture is real and mostly working — but the system is still in the awkward stage where there is no schema, no provenance, two inboxes, two Vault Keeper paths, half-finished widgets, and an Event Log that nobody writes to.

Read this as a constitution problem, not a feature problem. The highest-leverage work is not adding more surface area; it is writing the rules, reconciling the duplicates, and finishing the dashboard that lets the platform inspect itself.

03

Problems

The system is not broken in one dramatic place. It is fuzzy in a lot of small places, which makes it harder to trust. The most important issues are the ones that prevent governance, provenance, and self-healing.

Issue Why it matters Best next move
No constitution There is no root-level SCHEMA.md that says which folders exist, who owns them, and what fields they require. Promote the Vault Keeper schema into a canonical root document.
Duplicate Vault Keeper Two competing daemon paths exist with different schedules and labels; the wrong one is running. Pick one canonical daemon and unload the other.
Two inboxes 0-Inbox/ and 3-Resources/Inbox/ both collect capture with no firm rule. Commit to a single capture contract and document it.
No provenance Agent-written notes do not say who wrote them or when they should be treated as agent-owned. Add agent: and source: frontmatter to writers.
Unused Event Log The intended nervous system exists as a file, but the daemons barely write to it. Make every daemon append meaningful events.
Daemon stdout drift Some watchers have stale logs; it is unclear whether they are idle or hung. Teach Vault Keeper to probe, not just timestamp-check.
Telegram is capture-only The phone can send context into the vault, but it is not yet a full remote control surface. Wire Telegram to the hub and add control commands.
Root-level cruft Loose files and backups at the vault root make the repository hard to trust at a glance. Archive or move them once the schema is written.
04

Opportunities

The good news: the architecture already points at the right answers. The missing work is mostly connective tissue — the kind that turns a capable pile of parts into a platform.

Opportunity Why it matters Shape
Universal capture surface One contract for anything dropped into the vault, with Vault Keeper routing it on a timer. Capture folder with a clear contract.
PI Cockpit widgets Obsidian becomes a live command surface for health, triage, briefing, and status. Small focused widgets, not more windows.
Event Log as memory A plain markdown feed makes the platform legible and auditable. Every daemon appends meaningful events.
Daily briefings Morning notes can summarize health, capture, and open work instead of starting blank. Auto-generated daily note.
Provenance queries Frontmatter unlocks Dataview-style filters for agent ownership and capture type. Structured note metadata.
Smart files Scratchpad-style watchers can generalize into reusable conversational surfaces. Drop a note, get an answer.

06

Open questions

What is the canonical Vault Keeper?

This decides the heartbeat schedule, the label, and which script becomes the system’s actual nervous system.

What is the inbox contract?

This decides whether capture is routed automatically, reviewed manually, or split by source into a clean capture zone.

Does code stay inside the vault?

If yes, the notes and the infrastructure share one world model; if no, the workspace becomes more fragmented.

Do we commit to the Event Log?

If yes, the platform finally remembers its own actions; if no, the log should be retired instead of half-used.

What should happen to the loose root files?

Archive them or move them once the schema defines the authoritative home for each kind of artifact.
