Goal: turn Telegram voice notes or dropped audio files into searchable, richly summarized Markdown notes inside the Thoth Obsidian vault.
Voice capture → structured memory

Telegram audio becomes an Obsidian summary note.

A bot receives a native Telegram voice note or uploaded audio file, downloads it, runs Whisper transcription, passes the transcript into an LLM, and writes a polished Markdown summary into the vault with backlinks, metadata, and the raw transcript preserved.

Two capture paths, one downstream pipeline.

The intake layer should accept both Telegram's native compressed voice-note format and ordinary audio files. After download, everything gets normalized into the same job shape.

Path A: Native Telegram voice recorder

  • User records a voice note in a DM with the bot, group, or channel where the bot can read messages.
  • Telegram sends a voice object, usually OGG/Opus.
  • Bot uses getFile and downloads the audio from Telegram's file endpoint.
  • Filename is generated from timestamp, chat, sender, and message ID.

Path B: Dropped audio file

  • User uploads MP3, M4A, WAV, OGG, or another supported file to the bot.
  • Telegram sends an audio or document object.
  • Bot validates MIME type and size, then stores the file in the same inbox.
  • Optional: run ffmpeg normalization before Whisper.

Channel note If using a Telegram channel as the source, the bot generally needs to be added to that channel with enough permissions to receive posted messages. A simpler first version is DM-to-bot, then optionally forward the finished note link back to a channel.

Markdown output shape.

The summary note should be useful even months later: a tight abstract, action items, decisions, references, and the raw transcript tucked below the fold.

Recommended note sections

  • YAML frontmatter: source, sender, Telegram message ID, audio path, transcript path, model, date, tags.
  • Executive summary: 3–6 bullet synthesis of the voice memo.
  • Structured takeaways: decisions, tasks, open questions, people/projects/tools mentioned.
  • Cleaned narrative: readable prose version of the memo.
  • Raw transcript: preserved for auditability and search.

Summarizer prompt skeleton

You are converting a dictated voice memo into an Obsidian note.

Rules:
- Preserve the speaker's intent.
- Do not invent facts, dates, names, tasks, or decisions.
- If the transcript is ambiguous, mark it as an open question.
- Output clean Markdown only.
- Include a concise title.
- Extract action items as checkboxes when present.

Risks and design decisions.

Decision
Start with DM-to-bot.
It is simpler and more reliable than trying to treat a channel as the primary capture interface. Add channel ingestion once the core pipeline is stable.
Risk
Telegram file size and format variance.
Handle voice, audio, and document separately, then normalize before Whisper.
Risk
LLM hallucination in summaries.
Use a strict prompt, preserve the raw transcript, and instruct the model to mark uncertainty instead of filling gaps.
Decision
Write raw transcript and summary as separate artifacts.
This keeps the final note readable while preserving source material for search, audit, and re-summarization.
Single-file artifact generated for Thoth. Open locally or send through Telegram as an HTML attachment.
